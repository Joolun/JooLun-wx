# 小程序登录与接口访问规范

## 1. 文档目的

本文档用于统一当前小程序端的登录态初始化和接口调用方式。

本文档聚焦以下约束：

- 请求头没有带 `third-session`
- 页面进入太快，接口在登录完成前抢跑
- 同一个页面在 `onLoad`、`onShow` 中重复发起首批请求
- 二开时误以为要给每个接口单独加登录守卫

当前项目采用以下访问方式：

- 先登录
- 拿到 `third-session`
- 再发依赖登录态的接口

## 2. 登录原理

## 2.1 登录入口

小程序登录依赖微信 `wx.login()`。

后端登录接口：

- `/weixin/api/ma/malluser/login`

登录成功后，后端会返回一个自定义 session，并由小程序写入全局：

```js
that.globalData.thirdSession = channelUser.sessionKey
that.globalData.channelUser = channelUser
```

## 2.2 `third-session` 的作用

小程序后续访问商城接口时，后端统一通过请求头中的 `third-session` 识别当前用户。

如果没有带这个值，后端拦截器会拒绝请求。

## 2.3 请求头自动注入

当前小程序请求封装位于：

- `joolun-wx-ma/utils/api.js`

请求时会自动带上：

- `app-id`
- `third-session`

示意代码：

```js
header: {
  'app-id': wx.getAccountInfoSync().miniProgram.appId,
  'third-session': getApp().globalData.thirdSession != null ? getApp().globalData.thirdSession : ''
}
```

说明：

- 页面代码不需要手动维护 `third-session`
- 页面代码只需要保证“调用接口前，`app.initPage()` 已完成”

## 3. 后端校验机制

后端通过 `ThirdSessionInterceptor` 统一拦截 `/weixin/api/**`。

校验规则：

- 请求头存在 `third-session`
- Redis 中能取到对应 session

常见返回码：

- `60001`
  session 过期
- `60002`
  session 为空

## 4. 当前项目约定

## 4.1 标准写法

页面中所有首批依赖登录态的接口，都必须在 `app.initPage()` 成功后再调用。

标准写法：

```js
app.initPage()
  .then(() => {
    // 接口调用
  })
```

如果一个页面首屏有多个接口：

```js
app.initPage()
  .then(() => {
    this.loadMallUser()
    this.orderCountAll()
    this.loadAfterSaleStats()
  })
```

## 4.2 为什么这样做

因为当前项目的请求封装本身已经会自动带 `third-session`，问题不在接口封装层，而在页面调用时机。

只要页面在 `app.initPage()` 之后再发请求，接口就能正常访问。

## 4.3 页面职责边界

当前规范下，页面与接口层职责如下：

- `api.js` 负责统一拼装请求头
- `app.initPage()` 负责准备登录态
- 页面负责控制依赖登录态接口的调用时机

## 5. `app.initPage()` 的职责

当前 `app.initPage()` 负责：

- 若 `thirdSession` 不存在，则触发登录
- 若 `thirdSession` 存在，则通过 `wx.checkSession()` 校验是否失效
- 若 session 已失效，则重新登录
- 通过并发复用避免多个页面同时触发重复登录

当前要求：

- 保留 `initPage()` 的 Promise 复用能力
- 页面等待 `initPage()` 后再发首批请求

## 6. 页面接入规范

## 6.1 `onLoad` 首屏接口

适用于页面首次进入需要拉用户数据、订单数据、购物车数据的场景。

示例：

```js
onLoad() {
  app.initPage()
    .then(() => {
      this.loadData()
    })
}
```

## 6.2 `onShow` 刷新场景

如果页面在 `onShow` 里也要刷新依赖登录态的数据，仍然需要先走 `app.initPage()`。

示例：

```js
onShow() {
  this.refreshPageSafely()
},

refreshPageSafely() {
  if (this.refreshPromise) {
    return this.refreshPromise
  }
  this.refreshPromise = app.initPage()
    .then(() => {
      return Promise.allSettled([
        this.loadMallUser(),
        this.loadOrderStats()
      ])
    })
    .finally(() => {
      this.refreshPromise = null
    })
  return this.refreshPromise
}
```

说明：

- 如果 `onLoad` 和 `onShow` 会连续触发，建议做页面级 Promise 复用

## 6.3 纯公共接口

如果某个接口本身不依赖登录态，例如纯配置类、纯商品展示类接口，可以不强制等 `initPage()`。

但只要接口实际走了 `/weixin/api/**` 且依赖当前用户身份，就必须先 `initPage()`。

## 7. 典型页面场景

当前这类规范主要适用于以下页面场景：

- 首页
  首页统计、售后统计、客服引导文案
- 我的页
  商城用户资料、订单统计、售后统计
- 商品详情页
  购物车角标
- 下单确认页
  默认收货地址、商城用户资料
- 资料页
  商城用户资料加载
- 公共授权组件
  商城用户资料回显

这些页面都属于“先 `app.initPage()`，后调接口”的典型场景。

## 8. session 过期处理

当前请求封装中，若接口返回 `60001`，小程序会：

- 清空 `globalData.thirdSession`
- `reLaunch` 当前页面

代码效果等价于：

```js
getApp().globalData.thirdSession = null
wx.reLaunch({
  url: getApp().getCurrentPageUrlWithArgs()
})
```

这样做的结果是：

- 页面会重新走登录流程
- 用户基本无感
- 不需要在每个页面手写 session 过期恢复逻辑

## 9. 新页面接入建议

后续新增页面时，建议按以下顺序开发：

1. 先判断页面接口是否依赖当前登录用户
2. 如果依赖，则首批请求统一放到 `app.initPage().then(() => {})`
3. 如果 `onLoad` 和 `onShow` 都会触发，则补页面级 Promise 复用
4. 不要在 `api.js` 继续追加新的接口级登录守卫

## 10. 最终结论

当前项目的小程序登录与接口访问规范可以概括为一句话：

> 页面先执行 `app.initPage()`，登录完成后再调用依赖登录态的接口。

标准写法如下：

```js
app.initPage()
  .then(() => {
    // 接口调用
  })
```

后续二开请统一遵循这条约定。
